Game Engines

This log doesn't apply to just game engines, it applies to abstraction in general, including integrated circuits, standard libraries, frameworks and such. Specifically, it applies to the pre-built column in my other log The Three Columns of Abstraction. I'll go into deeper detail on the pros and cons of each extreme, and the situations in which I would use a game engine instead of programming from scratch.

Pros and Cons of a Game Engine

Pros: Easier to use, easier to prototype in, lower barrier of entry, pre-built advanced 3-D technologies usable from the start, better results for beginners.
Cons: More restrictive, imposes a framework on you, unremovable overhead (bigger size, more memory and cycles required), may be vendor-locked.

Pros and Cons of programming your own

Pros: Much more flexible, customized framework and workflow, no overhead (smaller size, less memory and cycles required), no licensing, full control.
Cons: Harder to get comparable results initially, tedious to prototype in, higher barrier of entry, education in 3-D technologies required.

Conclusion

Personally, I would default to building my own.

I would use an engine when...

  • I'm prototyping and I just want something out quickly
  • My boss tells me to
  • I don't really care about a commercial product and just want to be over it as fast as possible
  • I'm in a hackathon
  • I'm an artist and I just want a game out without doing any actual programming
  • I'm starving and I need money

I would program my own when...

  • I'm making a game with no deadlines
  • I'm learning how games and graphic technologies work
  • I'm making a unique game that is different technologically
  • I'm making a game that I'm passionate about and want it as good as possible
  • I'm making a game for a low-power device
  • I'm pushing the limits of the existing game engines
  • I want to have fun doing a career I love. ♠
