The Three Columns of Abstraction

You can script a game engine in assembly using a steady hand and a magnetic needle. You can write to the BIOS using Python and a hex editor. You can also write to the bare metal using assembly and a very sophisticated IDE. They are unrelated, and each of them form a column in the Three Columns of Abstraction.

The columns are:

Pre-built code

At the low end, you have bare metal*. At the high end, you have game engines.
Consequences of going low: Harder, more tedious BUT faster, smaller, more flexible.
Consequences of going high: Slower, bulkier BUT easier, less tedious.

Language

At the low end, you have machine language. At the high end, you have scripting languages.
Consequences of going low: Harder, more tedious BUT faster, smaller, more flexible.
Consequences of going high: Slower, bulkier BUT easier, less tedious.

Tooling

At the low end, you have a steady hand and a magnetic needle. At the high end, you have sophisticated IDEs.
Consequences of going low: Harder, more tedious BUT more flexible.
Consequences of going high: Easier, less tedious.

Conclusion

The combination that I go with personally is:

  • Pre-built code: Go as low as you practically can, for faster and smaller code
  • Language: Go as low as you practically can, for faster and smaller code
  • Tooling: Go as high as you practically can, for easier programming with no impact on software performance ♠
